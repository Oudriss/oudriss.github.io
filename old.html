Ceci à invoquer la notion de la conteneurisation qui a pu d'une manière ou autre d'isoler une application dans un conteneur qui ne dépend pas à l'environnement. Les conteneurs dans le monde informatique ne sont pas différents que celles du monde réel ce qu'on appelle les conteneurs intermodaux, le rôle de ces derniers est de transporter des biens ou des produits indépendamment du moyen de transport, aussi les conteneurs informatiques sert à livrer ou déployer une solution informatique indépendamment de l'environnement.

La conteneurisation


Aspet fonctionnel des conteneurs

Comme j'ai évoqué en introduction, les conteneurs du monde informatique cherchent à répondre à la même problématique que les conteneurs physiques. De très nombreux incidents survient lors de la mise en exploitation d'un développement afin de livrer une solution logicielle et c'est justement ce que la notion de conteneurisation cherche à résoudre.

Alors pour bien définir qu'est ce qu'un conteneur fonctionnellement, c'est une enveloppe virtuelle qui permet de packager une application avec tous les éléments dont elle a besoin pour fonctionner (fichiers source, runtime, librairies, outils et fichiers). Ils sont packagés en un ensemble cohérent et prêt à être déployé sur un serveur et son système d'exploitation (OS). Contrairement à la virtualisation, un conteneur n’embarque pas d’OS, il s’appuie directement sur celui du serveur sur lequel il est déployé. Un conteneur est donc beaucoup plus légé qu’une VM qui intégrera un OS. Une VM se mesure en Go quand le volume d’un conteneur se chiffrera en Mo.

Dans une architecture à base des conteneurs, le système d'exploitation contient les dépendances nécessaire pour faire fonctionner l'application déployer dans le conteneur. Par contre, l'application est géré par les développeurs de la solution logicielle. Ce qui offre une isolation qui permet aux développeurs d'embarquer l'ensemble des dépendances dont il besoin.

Aspet technique des conteneurs

Les système d'exploitation du hôte qui gére les conteneurs considère ces derniers comme des processus normaux dont il manipule les ressources (RAM, Stockage et processeur). Pour bien isoler ces processus, on se base sur les techniques suivantes :

  • CGroups : c'est une fonctionalité du noyau Linux a comme objectif la limiatation et l'isolation de l'utilisation des ressources.
  • Namespaces : cette fonctionalité permet à des porcessus de ne pas voir les ressources utilisées par d'autres, elle apporte l'isolation nécessaire à la création des fichiers des conteneurs
Donc, un conteneur est tout simplement un ensemble de fichier avec des processus qui s'éxecutent (conteneur) : Grâce à CGroups on limite les ressources et Namespace aide à isoler ces processus.

Docker et la conteneurisation

Docker est un projet Open Source proposant une surcouche qui automatise et simplifie le déploiement d’applications dans des conteneurs virtuels. A l’origine basée sur le format de Conteneurs Linux LXC, la société Docker a élargi le projet en proposant notamment une API qui permet d’exécuter des conteneurs standards et portables d’un serveur Linux à l’autre. Cette plateforme va vous permettre d'exécuter votre code à l'intérieur d'un conteneur indépendamment de la machine sur laquelle vous êtes.

Docker a apporté des nouveauté decisives aux conteneurs informatique, par exemple ils ont arrivé à optimiser la taille des conteneurs en mutualisant les données ainsi que plusieurs notions qui lui a permet d'être parmi les leaders.

Notion d'image

Comme nous avons vu un conteneur est un ensemble des fichiers sur lequel s'exécutent des processus. On peut alors construire une arborescence des fichiers dans OS qui pourra être les fichiers d'un conteneur. Alors, les images sont des archives des fichiers des conteneurs déjà crées qui peuvent être échangé et réutiliser pour créer d'autres conteneurs qui ont la même structure que le premier.

Docker Hub et registry

Le registry est l'annuaire et le centre de stockage des images (archives) réutilisables. Sa principale instanciation est le Docker Hub publique : Cliquer ici pour ouvrir le docker hub

Copy on write

Un conteneur peut se baser sur plusieurs images ou plutôt une pile d'images dont on ne peut pas modifier les fichiers de configuration. Pour ce faire, le fichier de configuration original sera copié et la nouvelle version surcharge la version originale. C'est le principe de Copy on write (COW) qui va nous permet de changer ou modifier une image déjà créée.

Docker engine

Le Docker Engine est le logiciel qui gère les conteneurs sur une machine, c'est lui qui gère les ressources utilisées par des conteneurs ainsi que les réseaux et le stockage.

Application multi-conteneur

Avec la notion des conteneurs et les apports ajoutés par Docker, on peut concevoir une architecture multi-conteneur pour fournir des services à exploiter.

Prenant l'exemple d'une application web avec une base de données MySQL, on a la possibilité de déployer l'application dans un conteneur et la base de données MySQL dans un autre conteneur. Allant plus loin, on ajoute un conteneur qui contient un serveur Nginx (Reverse proxy) qui va distribuer les requêtes sur trois instances du conteneur de l'application web, ces derniers interagissent avec un seul conteneur de MySQL pour des éventuels appels au base de données.

Docker nous permet de concevoir une tel architecture tout en bénéficiant de ces points fort tel que la taille des conteneurs et la gestion du stockage. Pour la communication entre les différents conteneur, Docker permet d'élaborer un réseau virtuel entre les conteneurs.

Docker swarm

Imaginer maintenant si on veut gérer plusieurs hôtes qui contient Docker afin de construire un Docker cluster. Cela peut être réalisé à l'aide du Docker swarm qui nous permet de gérer plusieurs hôtes contenant un Docker engine comme un seul hôte.

Pour cela, Docker swarm se base sur les points suivants :

  • Un client Docker : Il gère les conteneur du cluster Swarm.
  • Un service de découverte : C'est pour centraliser les informations sur tous les hôtes appartenant au Cluster.
  • Un maître de Swarm : ou bien Master, il gère les différents nœuds du cluster et il orchestre l'ensemble des conteneurs.
  • Plusieurs nœuds Swarm : l'ensemble des hôtes qui contient les conteneurs du Cluster Swarm.



Docker vs Virtualisation


Après avoir consulter plusieurs articles et tweets concernant le débat de docker contre la virtualisation, j'ai pu synthétiser ce débat dans les points suivants :
  • Docker est une manière de virtualisation mais au niveau système d'exploitation, par contre la vrai virtualisation se base sur des ressources physiques.
  • Docker conteneurs sont beaucoup plus léger que les VMs.
  • Docker n'embarque pas un système d'exploitation dans la machine, à l'opposé de la virtualisation.
  • La virtualisation nécessite plus de ressources (RAM et Stockage) pour construire des VMs, au contraire Docker s'appuie sur les ressources de système d'exploitation du hôte.
  • Les conteneurs ont des dépendances avec l'OS du hôte, par contre la virtualisation ne dépend pas à la machine hôte.

Conclusion et critiques


Certes, Docker a pu résoudre plusieurs problèmes en se basant sur la conteneurisation, il a pu découpler les applications et leurs environnement de déploiement afin de rendre ce dernier plus facile.
En revanche, même avec les avantages du Docker, nous ne pouvons pas négliger ces points faibles qui peuvent être un obstacle devant son utilisation.

Premièrement, avec les conteneurs Docker a pu isoler les applications de leurs environnements mais non pas ces dépendances avec l'environnement. En effet, la mise à jours du système de l'environnement peut influencer Docker lui-même, alors les conteneurs aussi peuvent être influencer, de ce fait Docker n'a pas pu isoler les applications à 100%.

Deuxièment, puisque les conteneurs sont basés sur des fichiers stockés dans le système d'exploitation de l'hôte, alors il y'a une possibilité que les conteneurs voient les fichiers d'autres conteneurs en changeant leurs nom d'espace (avec l'utilisateur ROOT) qui va causer des problèmes de sécurité.

Troisièment, avec des architectures complexes, la structure des fichiers des conteneurs devienne compliquée et désordonnée qui va rendre la gestion et la manipulation des conteneurs difficiles.

Finalement, grâce au Docker plusieurs technologies ont été développé d'une manière inimaginable, donnant l'exemple des Microservices et l'intégration continue dans la philosophie DevOps. Alors au niveau de ces dernières technologies, je préfère l'utilisation du Docker puisque ces avantages vont nous aider à bien adopter ces technologies. Par contre, dans les cas où nous voulons concevoir une architecture des applications de calculs ou bien un cluster de calcule, l'utilisation de la virtualisation est mieux que les conteneurs.
